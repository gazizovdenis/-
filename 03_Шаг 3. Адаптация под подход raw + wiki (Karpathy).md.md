# Шаг 3. Адаптация под подход raw + wiki (Karpathy)

`CLAUDE.md` есть. `Хранилище/` наполнено материалами. Сейчас разделим его на два слоя — это даст ИИ читать быстрее, не путаться, когда файлов станет много, и фиксировать связи между темами.

Эта схема — адаптация подхода Andrej Karpathy. Принцип простой: ИИ работает не с сырыми файлами напрямую, а с собранными по ним обзорами; источники цитируются явно; все операции пишутся в журнал.

---

## Зачем разделять

Когда в `Хранилище/` 10 файлов — ИИ читает всё подряд и отвечает нормально. Когда становится 200 — он начинает тонуть в деталях: открывает старый транскрипт вместо актуального обзора, путает цифры из разных встреч, тратит много токенов на чтение.

Решение — разделить материалы на **два слоя**.

| Слой | Что | Кто читает | Когда обновляется |
|------|-----|------------|-------------------|
| `raw/` | Сырые оригиналы как есть | ИИ — только если нужны детали | Только добавляем новые файлы |
| `wiki/` | Собранные обзоры (один файл — одна тема) | ИИ читает первым | Свободно переписываем |

ИИ заходит в `wiki/<тема>.md` за общей картиной (статус, команда, цифры, дедлайн) и только если нужно глубже — открывает конкретные файлы в `raw/`. Получается как с книгой: оглавление и краткое содержание сначала, полный текст потом.

---

## Назначение wiki

Wiki — это структурированная, связанная между собой база знаний для работы с ИИ.

- **ИИ ведёт wiki**: создаёт страницы по новым источникам, поддерживает их актуальность, ставит cross-links.
- **Вы курируете источники**: складываете файлы в `raw/`, задаёте вопросы, направляете анализ.

Это разделение труда: вы добавляете контекст, ИИ его структурирует.

---

## Структура

```
Хранилище/
├── raw/                          source documents (immutable, never modify)
│   ├── README.md                 мастер-индекс подпапок
│   ├── 00_Мусорка/
│   ├── 01_<проект>/
│   ├── 02_<проект>/
│   ├── 08_Личное/
│   ├── 95_Текущие_задачи/
│   └── 99_Остальное/
│
└── wiki/                         markdown pages maintained by Claude
    ├── index.md                  table of contents
    ├── log.md                    append-only journal of all operations
    ├── Портрет.md
    ├── _AI_WORKFLOW.md           (опционально) decision tree
    └── <тема>.md                 по странице на каждую тему / проект
```

Два служебных файла, без которых wiki не работает:

- `wiki/index.md` — карта всего, ИИ открывает её первой.
- `wiki/log.md` — журнал всех изменений. Append-only: только дописываем, ничего не редактируем.

## Миграция: из плоского Хранилища в raw + wiki

Делается за один вечер. Шесть шагов:

### 1. Создать подпапки raw/ и wiki/

```bash
cd <путь>/Система/Хранилище
mkdir -p raw wiki
```

### 2. Переместить всё текущее наполнение в raw/

Всё, что сейчас лежит в `Хранилище/` — это сырые материалы. Переносим внутрь `raw/`:

```bash
mv 0* 95_* 99_* raw/
mv 00_Мусорка raw/
```

### 3. Создать wiki/log.md (журнал)

```markdown
# Журнал операций wiki

Append-only. Каждая операция = одна запись.

Формат: `| YYYY-MM-DD | действие | объект | причина / источник |`

| Дата | Действие | Объект | Причина |
|------|----------|--------|---------|
| 2026-05-12 | created | wiki/ | миграция Хранилища на схему raw + wiki |
```

### 4. Собрать wiki/Портрет.md

Один файл, который ИИ открывает на любой вопрос про вас лично. Шаблон ниже.

### 5. Собрать wiki/<тема>.md на каждый активный проект

По одному файлу на проект. Шаблон ниже.

### 6. Собрать wiki/index.md

Карта всех wiki-страниц с таблицей «тип запроса → стартовый файл». Шаблон ниже.

### 7. Обновить CLAUDE.md

В раздел «Навигация» добавляем новые маршруты через `wiki/`. Скажите Claude: «обнови раздел Навигация в CLAUDE.md под структуру raw + wiki».

## Ingest workflow — как ИИ принимает новый источник

Когда вы добавляете новый файл в `raw/` (транскрипт встречи, договор, заметку) и говорите ИИ «обработай», он действует так:

1. **Читает полностью** новый файл.
2. **Обсуждает с вами ключевые тезисы** перед тем как что-то записать. Это критично: одно неверно понятое слово в источнике потом тиражируется по всем wiki.
3. **Создаёт страницу-обзор** в `wiki/` с именем по источнику. Например, источник `raw/02_<проект>/Встречи/2026-05-12_созвон.md` → wiki-страница `wiki/2026-05-12-sozvon.md` или обновление существующей `wiki/<проект>.md`.
4. **Создаёт или обновляет страницы по каждой ключевой сущности**: упомянуты новые клиенты / люди / решения → отдельные страницы или дополнения в существующие.
5. **Ставит cross-links** через `[[имя-страницы]]` между всеми связанными wiki-файлами.
6. **Обновляет `wiki/index.md`** — добавляет новые страницы с одной строкой описания.
7. **Дописывает запись в `wiki/log.md`**: дата, источник, что изменилось.

Один источник может затронуть 10–15 wiki-страниц. Это нормально.

## Page format — формат каждой wiki-страницы

Каждая страница в `wiki/` должна следовать этой структуре. Без неё ИИ не поймёт, насколько данные актуальны и откуда взяты.

```markdown
# <Название страницы>

**Summary:** 1–2 предложения, описывающие тему страницы.

**Sources:**
- raw/0X_<проект>/<файл-1>.md
- raw/0X_<проект>/<файл-2>.md

**Last updated:** YYYY-MM-DD

---

Основной текст. Чёткие заголовки, короткие абзацы.

Ссылки на связанные темы через [[wiki-links]] прямо в тексте — там, где упоминаются.

## Related pages

- [[связанная-страница-1]]
- [[связанная-страница-2]]
```

---

## Citation rules — правила цитирования

- Каждый факт в wiki должен ссылаться на источник.
- Формат после факта: `(source: filename.md)`.
- Если два источника противоречат друг другу — отметить противоречие явно, не выбирать в тихую.
- Если у факта нет источника — пометить `(source: needs verification)`.

Пример:

```markdown
Команда состоит из 3 человек (source: 2026-05-12_созвон.md).
Бюджет проекта на квартал — 500 тыс (source: 2026-04-30_встреча.md).
NSM: количество первичных клиентов из онлайн-каналов (source: needs verification).
```

## Question answering — как ИИ отвечает на ваши вопросы

Когда вы спрашиваете ИИ про что-то из системы:

1. Сначала открыть `wiki/index.md` и найти релевантные страницы.
2. Прочитать эти страницы, синтезировать ответ.
3. В ответе сослаться на конкретные wiki-страницы.
4. Если ответа в wiki нет — честно сказать «не нашёл в wiki, есть только в raw: <путь>».
5. Если ответ получился ценный — предложить «сохранить это как wiki-страницу?».

Хорошие ответы возвращаются в wiki, чтобы накапливаться, а не повторяться от чата к чату.


```markdown
# Журнал операций wiki
```

## Lint — аудит wiki

Когда система разрастётся (10+ wiki-страниц), периодически просите ИИ:

```
Проведи lint wiki:
- найди противоречия между страницами
- найди страницы без входящих ссылок (orphan)
- найди упоминания концепций, у которых нет своей страницы
- отметь устаревшие утверждения (есть более свежий source)
- проверь что все страницы следуют формату Summary / Sources / Last updated
- выдай отчёт нумерованным списком с предложенными правками
```

ИИ просканирует `wiki/` и вернёт пронумерованный список «проблема → как починить». Каждую правку можно применить отдельной командой.

## Правила wiki (обязательно)

### W1. raw — неизменяемое хранилище

- **Содержимое** файлов в `raw/` никогда не меняется. Это первоисточники.
- Можно: добавлять новые файлы, переименовывать папки, создавать подпапки для структуры (не контента).
- Исключения — папки которые редактируются: `raw/08_Личное/`, `raw/95_Текущие_задачи/`, `raw/00_Мусорка/`.

### W2. ИИ никогда не удаляет файлы

- Ни из `raw/`, ни из `wiki/`, ни из любого другого места системы.
- Если файл выглядит как мусор / дубль / устаревший — переместить в `raw/00_Мусорка/` с сохранением исходного пути внутри.
  Например, файл из `raw/99_Остальное/X.md` идёт в `raw/00_Мусорка/<категория>/99_Остальное/X.md`.
- Перед каждым переносом — запись в `wiki/log.md` с обоснованием.
- Пользователь сам удаляет файлы из Мусорки вручную, когда сочтёт нужным.
- `raw/00_Мусорка/` — единственное место, куда можно перемещать файлы для «удаления».

### W3. Обновлять index.md и log.md после каждого изменения

- После создания / обновления / переноса любой wiki-страницы — дописать в `log.md`.
- После добавления новой темы — добавить в `index.md`.

### W4. Именование wiki-страниц

- Нижний регистр, через дефис: `medcentre-marketing.md`, `lichnoe-zdorovye.md`.
- Исключения для главных файлов: `Портрет.md`, `index.md`, `log.md`, `_AI_WORKFLOW.md` (нижнее подчёркивание — чтобы был в начале списка).
- Кириллица в именах допустима, но если планируете отдавать команде или хранить в git — используйте латиницу.

### W5. Создание новых папок в raw/

- Создавать новые подпапки внутри проектов только если они объединяют **минимум 10 файлов**.
- Если файлов меньше 10 — оставить в текущей папке и описать в `README.md`: какие файлы и о чём.
- Новые папки называть по системе: `01_`, `02_`, и т.д. (уже принятая нумерация).
- `README.md` обязателен в каждой папке: перечисляет содержимое, если нет подпапок.

### W6. Citation rules (повторно для строгости)

- Каждый факт в wiki ссылается на источник: `(source: filename.md)`.
- Противоречия между источниками — фиксировать явно.
- Утверждения без источника — пометить `(source: needs verification)`.

### W7. Когда непонятно — спросить

- Когда непонятно, как категоризовать новую тему — спросить.
- Когда два источника противоречат друг другу — спросить, какой считать актуальным.
- Когда непонятно, объединять две страницы в одну или оставлять раздельно — спросить.

### W8. Язык wiki

- Отвечать в чате и в wiki по умолчанию — на русском.
- Технические термины и имена файлов — можно на английском / латиницей.

## Продвинутая опция: decision tree в wiki/_AI_WORKFLOW.md

Если у вас больше 5 проектов и больше 30 wiki-страниц — добавьте отдельный playbook:

```markdown
# _AI_WORKFLOW — playbook AI

## Главный принцип

Все ответы — через систему. Если в `wiki/` или `raw/` нет ответа — честно сказать «не нашёл в `<путь>`», задать 1–2 вопроса.

## Decision tree

Запрос → что открыть:

- Про проект → `[[<проект>]]` → `raw/0X_<проект>/README.md` → `Заметки_чат.md`
- Про меня лично → `[[Портрет]]` → `raw/08_Личное/`
- Что сейчас в работе → `[[index]]` → `raw/95_Текущие_задачи/`
- Куда положить новый файл → `raw/README.md`
- Не нашёл в wiki → открыть `raw/README.md` → если нет → задать 1–2 вопроса

## Workflow обновления

Когда пользователь сообщает новый факт / решение / результат:

1. Определить тип:
   - Личный факт → `raw/08_Личное/<категория>/`
   - Проектное решение → `raw/0X_<проект>/Заметки_чат.md`
   - Новая задача → `raw/95_Текущие_задачи/`
2. Обновить соответствующий файл.
3. Если факт затрагивает wiki — обновить wiki-страницу + добавить запись в `log.md`.
4. Сообщить кратко: «Записал в `<путь>`: <что записал>».
```
